## Problem Statement

Manual image data labeling is slow, costly, and inconsistent for small ML teams in custom domains. Pretrained models (COCO YOLO, HuggingFace pipelines) miss custom classes, so cold-start labeling is brutal. Free tools (Label Studio, CVAT) lack out-of-the-box active learning loops; commercial tools (Roboflow, Encord) require cloud upload + monthly fees that solo ML engineers / 1-3 person startup teams can't justify.

The pitch: "Label 50, auto-label 500." A self-hosted, offline, free active-learning labeling tool.

## Demand Evidence

**None yet — pure thesis.** No user interviews, no waitlist, no paying customers, no labeler has been observed using a prototype. This is the single largest risk in this design.

What the founder believes (unverified):
- Label Studio and CVAT users want active learning but find the ML-backend setup too painful.
- Solo ML engineers at small startups can't afford Roboflow ($5K+/yr enterprise tier).
- A zero-config local active-learning tool would close that gap.

## Status Quo

Target user (chosen, not interviewed): solo ML engineer at a 1-3 person ML startup in a vertical-data domain (medical imaging, robotics perception, defense, manufacturing QA).

Hypothesized current workflow:
- Manually labels images in CVAT or Label Studio
- Writes custom Python scripts to bootstrap labels from pretrained models
- Runs notebooks to fine-tune YOLO/Detectron2 on partial data
- Repeats label → train → predict → review loop manually
- Burns ~60% of week on data work vs. modeling

**This workflow is hypothesized, not observed.** Status Quo section is the second-largest risk after Demand Evidence.

## Target User & Narrowest Wedge

Target: Solo ML engineer, 1-3 person ML startup, vertical-data domain, tools budget ~$0-100/mo, comfortable running local Python services.

Narrowest wedge (current MVP design): zero-config active-learning loop for image bbox labeling. Single command install. Manual label 50 → background fine-tune → predict rest → confidence-sorted review queue → export YOLO.

**Wedge sharpness untested.** Real labelers may want different things (export format support, multi-annotator review, NER + bbox combo, etc.) — we don't know.

- Offline / self-hosted requirement (no cloud dependency)
- Permissive license stack only (no AGPL — skip YOLOv8)
- CPU-friendly defaults (no GPU required for MVP demo path)

## MVP Scope (2 weeks)

In-scope:
- Upload image folder via web UI or CLI
- Manual bbox labeling (~50 seed samples) via React + Konva canvas
- Single-class detection only (multi-class deferred)
- YOLOv5n fine-tune triggered as FastAPI BackgroundTask
- Auto-predict unlabeled images with confidence scores
- Confidence-sorted review queue (low-confidence first)
- Keyboard shortcuts: J/K (next/prev), A (approve), E (edit), R (reject)
- Retrain trigger on review batch completion
- Export YOLO format only
- SQLite + local filesystem storage
- Single-user (no auth)

Out of scope for MVP:
- Multi-user / auth
- NER / text labeling
- COCO / Pascal VOC / JSONL export
- Postgres / Celery / Redis
- Audit log / IAA metrics
- Multi-class detection
- Active learning beyond uncertainty sampling

## Tech Stack

| Layer | Choice | Why |
|-------|--------|-----|
| Backend | FastAPI + BackgroundTasks | Async, no Celery/Redis needed at this scale |
| CV model | YOLOv5n (Apache-2.0) | Skip YOLOv8 AGPL trap; YOLOv5n CPU-friendly |
| Active learning | Custom uncertainty sampling (entropy of confidence dist) | modAL is overkill for single-loop MVP |
| Frontend | React + Konva.js | Bbox canvas — Streamlit too slow for >50 samples/min review |
| DB | SQLite via SQLAlchemy | Easy local; SQLAlchemy keeps Postgres path open |
| Image store | Local filesystem | No MinIO until multi-user |
| Job queue | FastAPI BackgroundTasks | No Celery; retrain async on same process |

## Open Questions / Reviewer Concerns

**Critical (block external launch until resolved):**
1. Will YOLOv5n actually produce useful predictions from 50 seed samples on the target user's vertical data? If 80% of users hit the "model didn't generalize" wall, the core pitch breaks. Mitigation: include a "guided cold-start" mode that auto-detects when seed is too small and asks for more samples.
2. Is the target persona (solo ML engineer at small startup) actually constrained by Label Studio's ML-backend setup, or by something else entirely? Unverified.
3. What's the differentiator vs. Label Studio with the `label-studio-ml-backend` package? That combo already does active learning, free, self-hosted.

**Important:**
4. License compatibility audit on all dependencies (HuggingFace models have varying licenses).
5. How does user get their data back if they want to switch tools? Export quality often more important than label-time UX.
6. Single-class limitation may kill most real use cases — most vertical-data domains have 3-15 classes. Consider scoping multi-class into MVP if it's <2 days of extra work.

**Soft:**
7. Distribution plan for the binary / pip package (see below).
8. Brand / name (currently "auto-label-studio" too close to existing Label Studio — trademark + SEO risk).

## Success Criteria

MVP demo success (week 2):
- Label 50 → train YOLOv5n → predict 500 → review → ≥70% of predictions auto-acceptable at 0.9 confidence on COCO-subset sanity dataset
- End-to-end flow takes <30 min on a laptop CPU
- Reviewer can process ≥200 samples/hour with keyboard shortcuts

External validation success (post-MVP, week 3-4):
- Show HN gets ≥50 comments and ≥20 GitHub stars in 48 hours
- At least 3 unsolicited "I tried it on my own data" reports from real ML practitioners
- At least 1 user reports the "Label 50 → auto-label 500" promise actually worked on their dataset

## Distribution Plan

- Open-source on GitHub under MIT license
- Single-command install: `pip install auto-label-studio && auto-label-studio start`
- Docker image as alternative: `docker run -p 8000:8000 ...`
- Launch channels: Show HN, /r/MachineLearning, /r/computervision, Twitter ML community, YC W24/S24 ML companies cold email
- CI: GitHub Actions → PyPI publish on tag + Docker Hub publish

**Pre-launch rename strongly recommended** — "auto-label-studio" collides with Label Studio's trademark and SEO. Suggest options: Loop, Quickref, Seedlabel, Tinylabel.
